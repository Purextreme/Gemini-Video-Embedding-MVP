# Gemini Embedding 2 视频检索最小验证项目

## 项目目标

本项目的唯一目标是：

**验证 `gemini-embedding-2-preview` 是否能够对“单镜头视频素材”生成有用的向量，并支持“用自然语言文本检索视频镜头”。**

这是一个 **最小可行验证（MVP）**，不是正式产品，不追求完整系统，不追求工程复杂度，不追求前端界面，不追求数据库架构。

---

## 当前前提

我会提供的素材具有以下特点：

- 所有视频都已经**提前切片完成**
    
- **一个视频文件只对应一个单独镜头**
    
- 不存在“多个不同镜头混合在同一个视频文件中”的情况
    
- 当前测试重点是：
    
    - 文本检索视频
        
    - 视频之间的相似性验证
        
    - 初步判断 Embedding 2 是否适合我的素材类型
        

因此，本项目 **不需要** 做以下事情：

- 不需要自动视频切片
    
- 不需要 shot detection
    
- 不需要长视频分段逻辑
    
- 不需要复杂的多阶段检索系统
    
- 不需要上线部署
    
- 不需要 Web UI
    
- 不需要向量数据库
    
- 不需要账号系统
    
- 不需要云端服务编排
    

---

## 核心问题

本项目只围绕以下几个核心问题展开：

1. 对于已经切好的单镜头视频，`gemini-embedding-2-preview` 能否生成稳定可用的 embedding
    
2. 是否可以通过自然语言描述，成功检索到对应的视频镜头
    
3. 检索结果的 Top-1 / Top-3 是否具有实际参考价值
    
4. 视频之间的相似度是否符合直觉
    
5. 这个方案是否值得后续继续扩展
    

---

## 项目边界

### 要做的事情

必须完成以下最小功能：

1. 读取本地目录中的视频文件
    
2. 为每个视频生成一个 embedding
    
3. 为输入的文本 query 生成 embedding
    
4. 计算文本向量与视频向量之间的相似度
    
5. 输出每个 query 的 Top-K 检索结果
    
6. 结果应当尽量可读、可复查、可人工判断是否命中
    

### 不要做的事情

除非我明确要求，否则禁止主动加入以下内容：

- 自动切片
    
- 自动抽帧可视化
    
- Web 页面
    
- Electron 应用
    
- React 前端
    
- Flask / FastAPI 服务
    
- 向量数据库（如 FAISS、Milvus、Pinecone、Weaviate、Chroma）
    
- 复杂配置系统
    
- Docker
    
- 云部署
    
- 多线程优化
    
- 任务队列
    
- 账号权限系统
    
- 复杂日志系统
    
- 过度抽象的工程架构
    
- 与项目目标无关的“最佳实践”堆砌

## 技术路线约束

本项目必须遵循 **最简单、最直接、最容易验证** 的路线。

### 推荐路线

- 使用 **Python**
    
- 使用官方 SDK：`google-genai`
    
- 使用模型：`gemini-embedding-2-preview`
    
- 本地运行
    
- 单脚本优先
    
- 先输出到终端
    
- 如需持久化，优先使用：
    
    - JSON
        
    - CSV
        

### 明确限制

- 优先保证“先跑通”
    
- 优先保证“结果能人工检查”
    
- 优先保证“代码短而清晰”
    
- 优先避免为了“工程感”引入复杂结构
    

---

## 输入数据约束

### 视频素材

输入视频满足以下条件：

- 已经是单镜头视频
    
- 常见格式优先：
    
    - `.mp4`
        
    - `.mov`
        
- 每个文件代表一个独立的检索单元
    
- 一个视频文件对应一个 embedding
    

### 文本查询

文本查询应当用于模拟真实检索需求，例如：

- 画面内容描述
    
- 镜头动作描述
    
- 构图描述
    
- 产品展示方式描述
    
- 场景描述
    
- 人物行为描述
    

查询的目标不是生成文学描述，而是尽可能接近真实搜索语句。

---

## 输出结果要求

程序输出必须能让我直接判断是否“搜对了”。

### 最低要求

对于每一个文本 query，至少输出：

- query 原文
    
- Top-1 结果
    
- Top-3 结果
    
- 对应相似度分数
    

### 推荐输出形式

可接受的输出形式：

1. 终端打印结果
    
2. 保存为 CSV
    
3. 保存为 JSON
    

### 结果示例

```text
Query: a hand rotating a silver product in close-up

Top 1:
  shot_012.mp4   0.8421

Top 3:
  shot_012.mp4   0.8421
  shot_007.mp4   0.8014
  shot_019.mp4   0.7742
```

## 成功标准

本项目不是以“代码多完整”为成功标准，而是以“是否验证核心假设”为成功标准。

### 视为有效验证的标志

如果出现以下情况，说明测试有价值：

- 多数 query 能把正确视频排进 Top-3
    
- 相同类型镜头之间相似度较高
    
- 不同类型镜头之间相似度明显拉开
    
- 检索结果整体符合直觉
    
- 可以初步看出这个 embedding 是否适合我的素材
    

### 不必追求的内容

当前阶段不需要追求：

- 学术级评测
    
- 大规模 benchmark
    
- 线上实时检索性能
    
- 海量素材扩展能力
    
- 工业级架构设计

## 模型执行约束

在帮助实现本项目时，模型必须始终聚焦于以下原则：

1. **核心目标只有一个：验证 Embedding 2 对单镜头视频检索是否有效**
    
2. **一切与该目标无直接关系的复杂设计都应当延后**
    
3. **优先提供最小可运行方案**
    
4. **优先减少用户作为非程序员的使用门槛**
    
5. **优先让结果容易观察、容易判断、容易复现**
    
6. **不要擅自扩展到完整产品形态**
    
7. **不要把“以后可能需要”的内容提前塞进当前 MVP**
    
8. **如无明确要求，默认用户现在只想先做一个本地实验**

## 当前阶段建议实现顺序

### Phase 1：最小验证版

实现以下能力即可：

- 读取本地视频文件
    
- 调用 Gemini Embedding 2 生成视频向量
    
- 调用同一模型生成文本向量
    
- 计算 cosine similarity
    
- 输出 Top-K 结果
    

### Phase 2：轻量增强版

仅在 Phase 1 跑通后，再考虑：

- 保存 embedding 到本地 JSON
    
- 保存 query 检索结果到 CSV
    
- 增加简单的批量 query 测试
    
- 增加人工标注的命中检查表
    

### Phase 3：是否继续扩展，再由我决定

例如未来可能扩展：

- 本地素材库管理
    
- 更大的数据集
    
- 向量索引
    
- 简单界面
    
- 与 transcript / metadata 融合
    

但这些都 **不是当前必须内容**。
